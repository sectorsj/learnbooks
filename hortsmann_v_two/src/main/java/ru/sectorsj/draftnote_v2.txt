// page 288
// 5.1 Структура JDBC (The Design of JDBC)

// Создатели Java с самого начала осознавали потенциальные преимущества данного языка
// для работы с базами данных.
// С 1995 года они начали работать над расширением стандартной библиотеки Java
// для организации доступа к базам данных средствами SQL.

// Сначала они попробовали создать такие расширения Java,
// которые позволили бы осуществлять доступ
// к произвольной базе данных только средствами Java,
// но очень скоро убедились в бесперспективности такого подхода,
// т.к. для доступа к базам данных применялись самые разные протоколы.

// Поставщики программного обеспечения баз данных
// были весьма заинтересованы в разработке на Java стандартного сетевого протокола
// для доступа к базам данных, при условии,
// что за основу будет принят их собственный сетевой протокол.

// В конечном счете:
// - поставщики баз данных
// - инструментальных средств для доступа к ним
// сошлись на том, что лучше предоставить прикладной интерфейс API только на Java
// для доступа к базам данных средствами SQL, а также диспетчер драйверов,
// который позволил бы подключать к базам драйверы независимых производителей.

// Такой подход позволял поставщикам баз данных создавать собственные драйверы,
// которые подключались бы с помощью данного диспетчера.

// Предполагалось, что это будет простой механизм регистрации сторонних драйверов.

// Подобная организация прикладного интерфейса JDBC
// основана на весьма удачной модели интерфейса ODBC,
// разработанного в корпорации Microsoft.

// В основу интерфейсов JDBC и ODBC положен общий принцип:
// программы, написанные в соответствии с требованиями прикладного интерфейса API,
// способны взаимодействовать с диспетчером драйверов JDBC,
// который, использует подключаемые драйверы для обращения к базе данных.

// Это означает, что для работы с базами данных в прикладных программах
// достаточно пользоваться средствами JDBC API.


// page 288
// 5.1.1 Типы драйверов JDBC (JDBC Driver Types)

// Каждый драйвер JDBC относится к одному из перечисленных типов:

// Драйвер:
// Тип 1: Преобразует интерфейс JDBC в ODBC.
//        Для взаимодействия с базой данных использует драйвер ODBC.
//        Один такой драйвер был включен в первые версии Java под названием мост JDBC/ODBC.
//        Но для его применения требуется установить и настроить соответствующим образом драйвер ODBC.
//        В первом выпуске JDBC этот мост предполагалось использовать только для тестирования,
//        а не для применения в рабочих программах.
//        В настоящее время уже имеется достаточное количество более удачных драйверов,
//        поэтому пользоваться мостом JDBC/ODBC не рекомендуется.

// Тип 2: Написан частично на Java и отчасти использует платформенно-ориентированный код
//        для взаимодействия с клиентским прикладным интерфейсом API базы данных.
//        Для применения такого драйвера, помимо библиотеки Java,
//        на стороне клиента необходимо установить код, специфический для конкретной платформы.

// Тип 3: Разрабатывается только на основе клиентской библиотеки Java,
//        в которой используется независимый от базы данных протокол передачи запросов базы данных на сервер.
//        Этот протокол приводит запросы базы данных в соответствие с характерным для нее протоколом.
//        Развертывание прикладных программ значительно упрощается благодаря тому,
//        что код, зависящий от конкретной платформы, находится только на сервере.

// Тип 4: Представляет собой библиотеку, написанную только на Java,
//        для приведения запросов JDBC в соответствие с протоколом конкретной базы данных.

// Спецификация прикладного интерфейса JDBC доступна для загрузки по адресу:
// https://jcp.org/en/jsr/detail?id=221

// Большинство поставщиков баз данных предоставляют драйверы 3 или 4 типа.
// Целый ряд сторонних производителей специализируется на создании драйверов,
// которые позволяют добиться более полного соответствия принятым стандартам,
// поддерживают большее количество платформ,
// обладают более высокой производительностью или надежностью, чем драйверы,
// предлагаемые поставщиками баз данных.

// Основные цели прикладного интерфейса JDBC:
// 1. Разработчики пишут программы на Java, пользуясь для доступа к базам данных
//    стандартными средствами языка SQL или его специализированными расширениями,
//    следуя только соглашениям, принятым в Java.

// 2. Поставщики баз данных и инструментальных средств к ним
//    предоставляют драйверы только низкого уровня.
//    Это дает им возможность оптимизировать драйверы под свою конкретную продукцию.


// page 290
// 5.1.2 Типичные примеры применения JDBC (Typical Uses of JDBC)

// Согласно традиционной модели "клиент-сервер" графический пользовательский интерфейс (ГПИ)
// реализуется на стороне клиента, а базы данных располагается на стороне сервера (рис. 5.1)
// (resources/_290/pic_5_1.JPG)

// В этом случае драйвер JDBC развертывается на стороне клиента.

// В настоящее время существует тенденция по переходу:
// от архитектуры "клиент-сервер"
// к трехуровневой модели (three-tier)
//  или
// n-уровневой модели

// В трехуровневой модели:
// клиент не формирует обращения к базе данных.
// Он обращается к средствам промежуточного уровня на сервере,
// который, выполняет запросы к базе данных.

// Трехуровневая модель, обладает преимуществами.
// Она отделяет:
// 1. визуальное представление (на клиентском компьютере)
// 2. бизнес-логику (на промежуточном уровне)
// 3. исходных данных (хранящихся в базе данных).

// -> Становится возможным доступ к тем же самым данным
// по одинаковым бизнес-правилам со стороны разнотипных клиентов,
// в том числе:
//  - прикладных программ на java
//  - веб-браузеров
//  - приложений для мобильных устройств

// Взаимодействие между клиентом и промежуточным уровнем
// может быть реализовано по сетевому протоколу НТТР.

// JDBC - прикладной интерфейс, который служит для управления взаимодействием между:
// - промежуточным уровнем
// - серверной базой данных

// На рис. 5.2 (resources/_290/pic_5_2.JPG)
// схематически показана основная архитектура трехуровневой модели.


// page 291
// 5.2 Язык SQL (Structured Query Language)

// Прикладной интерфейс JDBC позволяет взаимодействовать с базами данных посредством языка SQL,
// который, образует интерфейс для большинства современных реляционных баз данных.

// Настольные базы данных предоставляют графический интерфейс,
// который дает пользователям возможность непосредственно манипулировать данными,
// но доступ к серверным базам данных возможен только средствами языка SQL.

// Пакет JDBC можно рассматривать лишь как прикладной интерфейс API
// для взаимодействия с операторами языка SQL с целью получить доступ к базам данных.

// который позволяет связывать данные из разных таблиц
// и получать более полную и осмысленную информацию.

// Это позволяет эффективно хранить данные в нормализованной форме (разделенными по таблицам)

// Преимущество соединения таблиц заключается в том,

// Например, в простейшей структуре баз данных таблица "Books"

// В реляционной модели данные распределяются среди нескольких таблиц таким образом,

// для создания запросов путем манипулирования столбцами и ввода данных в готовые формы.

// Они называются инструментальными средствами составления "запросов по образцу" (QBE) (query by example).
// А при использовании SQL запрос создается в текстовом виде в строгом соответствии
// с синтаксисом этого языка, как показано ниже:
    SELECT Books.Title Books.Publisher_Id, Books.Price, Publishers.Name, Publishers.URL

    WHERE Books.Publisher_Id = Publisher.Publisher_Id

    SELECT * FROM Books;

// в которых требуется выполнить поиск данных.

// В операторе SELECT можно указать любые требующиеся столбцы следующим образом:
    SELECT ISBN, Price, Title
    FROM Books;

// Выбор строк можно ограничить с помощью условия, указываемого в предложении WHERE:
    SELECT ISBN, Price, Title

// 5.4 Работа с операторами JDBC (Working with JDBC Statements)

// Как пользоваться классом Statement из прикладного интерфейса JDBC для:
// - выполнения операторов SQL,

// page 302
// 5.4.1 Выполнение операторов SQL (Executing SQL Statements)

// Для выполнения оператора SQL:
// 1. Создается объект типа Statement.
//    Для этой цели используется объект типа Connection,
//    который можно получить, вызвав метод DriverManager.getConnection() следующим образом:
        Staternent stat = conn.createStaternent();

// 2. Формируется символьная строка с требующимся оператором SQL:
        String command = "UPDATE Books"
        + " SET Price = Price - 5.00"
        + " WHERE Title NOT LIKE '%Introduction%'";

// 3. Вызывается метод executeUpdate() из класса Statement:
        stat.executeUpdate(command);

// Метод executeUpdate() - возвращает количество строк, полученных из таблицы базы данных
//                         в результате выполнения оператора SQL, или же нуль строк для тех операторов,
//                         которые не возвращают количество строк из таблицы.

// -> вызов метода executeUpdate() возвращает количество книг, цена которых снижена на 5 долларов.

// Вызывая метод executeUpdate(), можно выполнять операторы:
//  - INSERT
//  - UPDATE
//  - DELETE
// а также операторы определения данных, в том числе:
//  - CREATE TABLE
//  - DROP TABLE

// Для выполнения оператора SELECT необходимо вызвать, другой метод executeQuery().

// execute() - универсалышй метод может выполнять произвольные операторы SQL.
//             Применяется для составления запросов в диалоговом режиме.

// Если составляется запрос для базы данных, будет интересен результат его обработки.

// Метод executeQuery() - возвращает объект типа ResultSet, который можно использовать
// для построчного просмотра результатов выполнения запроса:
    ResultSet rs = stat.executeQuery("SELECT * FROM Books")

// Для анализа результирующего набора организуется простой цикл:
    while (rs.next()) {
        // проанализировать строку из результирующего набора
    }

// Порядок последовательной обработки строк в интерфейсе ResultSet
// организован несколько иначе, чем в интерфейсе java.util.Iterator.

// В интерфейсе ResultSet итератор устанавливается на позиции
// перед первой строкой из результирующего набора.
// Поэтому для его перемещения к первой строке нужно вызвать метод next().

// В интерфейсе ResultSet отсутствует метод hasNext(),
// -> метод next() придется вызывать до тех пор,
// пока не будет возвращено логическое значение false.

// В результирующем наборе строки располагаются в произвольном порядке.
// Если порядок их следования важен, его необходимо установить с помощыо предложения ORDER ВУ.

// При обработке каждой строки таблицы
// требуется получить содержимое отдельных полей (или столбцов).

// Для этой цели имеется целый ряд методов доступа к полям (или столбцам).

// Примеры вызова подобных методов доступа:
    String isbn = rs.getString(1);
    String price = rs.getDouble("Price");

// Для каждого типа данных Java предусмотрен отдельный метод доступа,
// - getString()
// - getDouble()
// - и др
// Каждый из них реализован в двух формах:
// одной - с числовым параметром,
// другой - со строковым.

// Если метод доступа вызывается с числовым параметром,
// данные извлекаются из столбца с указанным номером.

// -> вызова метода rs.getString(1):
// возвращается значение из первого столбца текущей строки таблицы.

// В отличие от массивов, нумерация столбцов таблиц в базе данных начинается с 1.

// Если же метод доступа вызывается со строковым параметром,
// данные извлекаются из столбца с указанным именем.

// -> вызова метода rs.getDouble("Price"):
возвращается значение из столбца с именем Price.

// Первая форма методов доступа с числовым параметром более эффективна,
// но строковые параметры улучшают восприятие и упрощают сопровождение кода.

// Если указанный тип не соответствует фактическому типу,
// метод доступа автоматически выполняет необходимое преобразование данных.

// -> вызова метода rs.getString("Price"):
// числовое значение с плавающей точкой, извлекаемое из столбца Price,
// преобразуется в символьную строку.


// page 305
// 5.4.2 Управленеи подключениями, операторами и результирующими наборами
//       (Managing Connections, Statements, and Result Sets)

// Каждый объект типа Connection может создать один или несколько объектов типа Statement.

// Один и тот же объект типа Statement можно использовать
// для выполнения нескольких не связанных между собой операторов SQL и запросов.
// Но для такого объекта допускается наличие не более oдногo открытого результирующего набора.

// Если же требуется выполнить несколько запросов и одновременно проанализировать их результаты,
// то для этого понадобится несколько объектов типа Statement.

// Нужно помнить про ограничения, накладываемые на количество операторов SQL,
// приходящихся на одно подключение к базе данных.

// Количество открытых объектов типа Statement, одновременно поддерживаемых драйвером JDBC,
// можно выяснить, вызвав метод getMaxStatements() из интерфейса DatabaseMetaData.

// Не стоит одновременно обрабатывать несколько результирующих наборов.

// Если же результирующие наборы (ResultSet) взаимосвязаны,
// следует выдать составной запрос и проанализировать единый результат.

// Намного эффективнее объединять запросы в самой базе данных,
// чем выполнять обход нескольких результирующих наборов в программе на Java.

// Примеры:
// 1. объединения запросов в базе данных
// 2. обхода нескольких результирующих наборов в Java

// 1. Объединение запросов в базе данных (на примере SQL):
// У нас есть две таблицы: "orders" (заказы) и "customers" (покупатели).
// Нужно получить информацию о заказах вместе с именами покупателей.

// sql запрос:
// SELECT orders.order_id, orders.order_date, customers.customer_name
// FROM orders
// JOIN customers ON orders.customer_id = customers.customer_id;

// В этом примере используется оператор JOIN
// для объединения таблиц "orders" и "customers" на основе поля "customer_id".
// Это позволяет получить информацию из обеих таблиц в одном запросе.

// 2. Обход нескольких результирующих наборов в Java:
// Было выполнено два отдельных запроса к базе данных
// и получено два результирующих набора:
// - "orders"
// - "customers".
// Теперь нужно объединить эти наборы в Java.

// java code:
// List<Order> orders = // получаем список заказов из базы данных
// List<Customer> customers = // получаем список покупателей из базы данных

// for (Order order : orders) {
//     for (Customer customer : customers) {
//         if (order.getCustomerId() == customer.getId()) {
//             order.setCustomerName(customer.getName());
//             break;
//         }
//     }
// }

// В этом примере используется вложенные циклы для обхода списков "orders" и "customers".
// Для каждого заказа, где ищется соответствующий покупатель по полю "customer_id"
// и устанавливаем имя покупателя в объекте заказа.

// Как можно заметить, объединение запросов в базе данных является более эффективным подходом,
// так как база данных оптимизирована для выполнения таких операций.
// Это позволяет уменьшить количество передаваемых данных и уменьшить нагрузку на приложение.

// В некоторых случаях может потребоваться выполнить обход результирующих наборов в Java,
// например, когда необходимо выполнить сложную бизнес-логику или преобразования данных,
// которые невозможно реализовать на уровне базы данных.

    class ExecSQL {
       public static void main(String args[]) throws IOException {
          try (Scanner in = args.length == 0 ? new Scanner(System.in)
                : new Scanner(Path.of(args[0]), StandardCharsets.UTF_8)) {
             try (Connection conn = getConnection();
                   Statement stat = conn.createStatement()) {
                while (true) {
                   if (args.length == 0)
                      System.out.println("Enter command or EXIT to exit:");

                   if (!in.hasNextLine())
                      return;

                   String line = in.nextLine().strip();
                   if (line.equalsIgnoreCase("EXIT"))
                      return;
                   if (line.endsWith(";")) // remove trailing semicolon
                      line = line.substring(0, line.length() - 1);
                   try {
                      boolean isResult = stat.execute(line);
                      if (isResult) {
                         try (ResultSet rs = stat.getResultSet()) {
                            showResultSet(rs);
                         }
                      } else {
                         int updateCount = stat.getUpdateCount();
                         System.out.println(updateCount + " rows updated");
                      }
                   } catch (SQLException e) {
                      for (Throwable t : e)
                         t.printStackTrace();
                   }
                }
             }
          } catch (SQLException e) {
             for (Throwable t : e)
                t.printStackTrace();
          }
       }

       public static Connection getConnection() throws SQLException, IOException {
          var props = new Properties();
          try (Reader in = Files.newBufferedReader(
                Path.of("database.properties"), StandardCharsets.UTF_8)) {
             props.load(in);
          }
          String drivers = props.getProperty("jdbc.drivers");
          if (drivers != null)
             System.setProperty("jdbc.drivers", drivers);

          String url = props.getProperty("jdbc.url");
          String username = props.getProperty("jdbc.username");
          String password = props.getProperty("jdbc.password");

          return DriverManager.getConnection(url, username, password);
       }

       public static void showResultSet(ResultSet result) throws SQLException {
          ResultSetMetaData metaData = result.getMetaData();
          int columnCount = metaData.getColumnCount();

          for (int i = 1; i <= columnCount; i++) {
             if (i > 1)
                System.out.print(", ");
             System.out.print(metaData.getColumnLabel(i));
          }
          System.out.println();

          while (result.next()) {
             for (int i = 1; i <= columnCount; i++) {
                if (i > 1)
                   System.out.print(", ");
                System.out.print(result.getString(i));
             }
             System.out.println();
          }
       }
    }

            // Обработка исключения SQLException
            for (Throwable t : e) {
                // Вывод информации об исключении и трассировки стека
                t.printStackTrace();
            }
        }
    }

            }
        }
    }

    private static void executeInsertStatement(Statement stat, String sql) throws SQLException {
        String[] lines = sql.split("\\r?\\n");
        StringBuilder sb = new StringBuilder();
        for (String line : lines) {
            line = line.trim();
            if (!line.isEmpty()) {
                if (line.endsWith(",")) {
                    sb.append(line).append(" ");
                } else {
                    sb.append(line);
                    executeStatement(stat, sb.toString());
                    sb.setLength(0);
                }
            }
        }
    }

// Полный код класса `ExecSQL` с учетом изменений для обработки скриптов
// с переносами строк и множественными значениями в одном операторе `INSERT INTO`:

import java.io.IOException;
import java.nio.file.Files;
import java.nio.file.Path;
import java.sql.*;
import java.util.Scanner;

public class ExecSQLv2 {
    public static void main(String[] args) throws IOException {
        try (Scanner in = args.length == 0
                ? new Scanner(System.in)
                : new Scanner(Path.of(args[0]))) {
            try (Connection conn = getConnection();
                 Statement stat = conn.createStatement()) {
                while (true) {
                    if (args.length == 0) {
                        System.out.println("Enter command, LOAD FILES to load data from files, or EXIT to exit:");
                    }
                    if (!in.hasNextLine()) {
                        return;
                    }
                    String line = in.nextLine().trim();
                    if (line.equalsIgnoreCase("EXIT")) {
                        return;
                    }
                    if (line.equalsIgnoreCase("LOAD FILES")) {
                        System.out.println("Enter the paths to the script files (separated by commas):");
                        String filePaths = in.nextLine().trim();
                        String[] paths = filePaths.split(",");
                        for (String path : paths) {
                            path = path.trim();
                            try {
                                String content = Files.readString(Path.of(path));
                                executeScript(stat, content);
                            } catch (IOException e) {
                                System.out.println("Warning: File not found or invalid path - " + path);
                            }
                        }
                    } else {
                        if (line.endsWith(";")) {
                            line = line.substring(0, line.length() - 1);
                        }
                        executeStatement(stat, line);
                    }
                }
            }
        } catch (SQLException e) {
            for (Throwable t : e) {
                t.printStackTrace();
            }
        }
    }

    private static Connection getConnection() throws SQLException {
        String url = "jdbc:derby:booksdb";
        return DriverManager.getConnection(url);
    }

    private static void executeScript(Statement stat, String script) throws SQLException {
        String[] statements = script.split(";");
        for (String sql : statements) {
            sql = sql.trim();
            if (!sql.isEmpty()) {
                if (sql.toUpperCase().startsWith("INSERT INTO")) {
                    executeInsertStatement(stat, sql);
                } else {
                    executeStatement(stat, sql);
                }
            }
        }
    }

    private static void executeInsertStatement(Statement stat, String sql) throws SQLException {
        String[] lines = sql.split("\\r?\\n");
        StringBuilder sb = new StringBuilder();
        for (String line : lines) {
            line = line.trim();
            if (!line.isEmpty()) {
                if (line.endsWith(",")) {
                    sb.append(line).append(" ");
                } else {
                    sb.append(line);
                    executeStatement(stat, sb.toString());
                    sb.setLength(0);
                }
            }
        }
    }

    private static void executeStatement(Statement stat, String sql) throws SQLException {
        try {
            boolean isResult = stat.execute(sql);
            if (isResult) {
                try (ResultSet rs = stat.getResultSet()) {
                    showResultSet(rs);
                }
            } else {
                int updateCount = stat.getUpdateCount();
                System.out.println(updateCount + " rows updated");
            }
        } catch (SQLException e) {
            for (Throwable t : e) {
                t.printStackTrace();
            }
        }
    }

    private static void showResultSet(ResultSet result) throws SQLException {
        ResultSetMetaData metaData = result.getMetaData();
        int columnCount = metaData.getColumnCount();

        for (int i = 1; i <= columnCount; i++) {
            if (i > 1) {
                System.out.print(", ");
            }
            System.out.print(metaData.getColumnLabel(i));
        }
        System.out.println();

        while (result.next()) {
            for (int i = 1; i <= columnCount; i++) {
                if (i > 1) {
                    System.out.print(", ");
                }
                System.out.print(result.getString(i));
            }
            System.out.println();
        }
    }
}

        + "FROM Books, Publishers "
        + "WHERE Books.Publishers_Id = Publishers.Publishers_Id "
        + "AND Publishers.Name = ?";
    PreparedStatment = stat = conn.prepareStatement(publisherQuery);

    stat.setString(1, publisher);

    }

// то из него можно получить символьные данные,
// вызвав метод:
// - String()
// или
// - getCharacterStream()

// Чтобы разместить большой объект в базе данных,
// 1. следует вызвать метод:
//    - createLOB()
//         или
//    - createClob()
//    для объекта типа Connection

// 2. получить
//    - поток вывода
//          или
//    - поток записи

// 3. записать данные

    Blob coverBlob = conn.createBlob();

// page 321
// 5.5.3 Синтаксис переходов в SQL (SQL Escapes)

// Синтаксис переходов (escape) предоставляет средства,
// которые обычно поддерживаются базами данных,
// но в разных вариантах в зависимости от конкретного синтаксиса базы данных.

// В задачу драйвера JDBC входит
// преобразование синтаксиса переходов
// в синтаксис конкретной базы данных.

// Переходы предусмотрены для следующих средств:
//  - Литералы времени и даты           (Date and time literals)
//  - Вызовы скалярных функций          (Calling scalar functions)
//  - Вызовы хранимых процедур          (Calling stored procedures)
//  - Внешние соединения                (Outer joins)
//  - Символы перехода в операциях LIKE (The escape character in LIKE clauses)

// Литералы даты и времени сильно отличаются в разных базах данных.

// Чтобы вставить литерал даты или времени, следует определить его значение в формате ISO 8601
// https://www.cl.cam.ac.uk/~mgk25/iso-time.html
// После этого драйвер преобразует литерал в собственный формат базы данных.

// 1. Для значений типа:
//    - DATE
//    - TIME
//    - TIMESTAMP

//    используются литералы:
//    - d
//    - t
//    - ts

//    следующим образом:
//    {d '2008-01-24'}
//    {t '23:59:59'}
//    {ts '2008-01-24 23:59:59.999'}

// 2. Скалярна функция - функция, которая возращает одно значение.
//    В базах данных применяется немало скалярных функций, но под разными именами.

//    В спецификации JDBC указаны стандартные имена,
//    преобразуемые в специфические имена, для конкретных баз данных.

//    Чтобы вызвать скалярную функцию, нужно вставить ее стандартное имя и аргументы:
       {fn left (?, 20)}
       {fn user()}

//    Полный список поддерживаемых имен скалярных функций можно найти в спецификации JDBC.

// 3. Хранимая процедура - процедура, которая выполняется в базе данных
//                         и написана на специальном языке для конкретной базы данных.

//    Для вызова хранимой процедуры служит переход call.

//    Если у процедуры отсутствуют параметры, то указывать скобки не нужно.
//    Для фиксации возвращаемого значения служит знак равенства.

//    Каким образом вызываются хранимые процедуры:
        {call PROC1(?, ?)}
        {call PROC2}
        {call ? = PROC3(?)}

// 4. Внешнее соединение (Outer join) двух таблиц не требует,
//    чтобы строки из каждой таблицы совпадали по условию соединения.

//    В приведенном запросе указаны книги, для которых столбец Publisher_Id
//    не имеет совпадений в таблице Publishers,
//    причем пустые значения NULL обозначают отсутствие совпадений.
        SELECT * FROM {oj Books LEFT OUTER JOIN Publishers
        ON Books.Publisher Id = Publisher.Publisher_Id}

// Чтобы включить в запрос издательства без совпадающих книг,
// может потребоваться предложение RIGHT OUTER JOIN,
// а чтобы возвратить по запросу и то и другое - предложение FULL OUTER JOIN.

// Синтаксис переходов требуется именно потому,
// что не во всех базах данных используется стандартное обозначение внешних соединений.

// 5. Знаки "_" и "%" имеют специальное назначение в операции LIKE,
//    обозначая совпадение с одним символом или последовательностью символов.

//    Стандартного способа их буквального употребления не существует.
//    Для сопоставления всех символьных строк, содержащих знак "_",
//    можно воспользоваться конструкцией:
        ... WHERE ? LIKE %!_% {escape '!'}
//    где, "!"  - знак определен как символ перехода,
//         "!_" - последовательность знаков, обозначает знак подчеркивания.


// page 323
// 5.5.4 Множественные результаты (Multiple Results)

// По запросу могут быть возвращены множественные результаты (Multiple Results).
// Это может произойти при выполнении хранимой процедуры или в базах данных,
// которые допускают также выполнение многих операторов SELECT в одном запросе.

// Получить все результирующие наборы можно следующим образом:
// 1. Вызвать метод execute() для выполнения оператора SQL.
// 2. Получить первый результат или подсчет обновлений.
// 3. Повторить вызов метода getMoreResults(), чтобы перейти к следующему результирующему набору.
// 4. Завершить процедуру, если больше не остается результирующих наборов или подсчетов обновлений.

// Методы:
// - execute()
// - getMoreResults()
// возвращают логическое значение true,
// если следующим звеном в цепочке оказывается результирующий набор.

// Метод get.UpdateCount() возвращает значение -1,
// если следующим звеном в цепочке не оказывается подсчет обновлений.

// В следующем цикле осуществляется последовательный обход всех полученных результатов:
         boolean isResult = stat.execute(command);
         boolean done = false;
         while(!done) {
             if (isResult) {
                 ResultSet result = stat.getResultSet();
                 // Сделать что-нибудь с полученным результатом в переменной result
             } else {
                 int updateCount = stat.getUpdateCount();
                 if (updateCount >= 0) {
                     // Сделать что-нибудь с подсчетом обновлений в переменной updateCount
                 } else{
                     done = true
                 }
             }
             if (!done) {
                 isResult = stat.getMoreResults();
             }
         }

// 5.5.5 Извлечение автоматически генерируемых ключей (Retrieving Autogenerated Keys)

// В большинстве баз данных поддерживается механизм автоматической нумерации строк в таблице.
// У разных поставщиков баз данных эти механизмы заметно отличаются.

// Автоматически присваиваемые номера часто используются н качестве первичных ключей.
// Несмотря на то что в JDBC не предлагается независимое
// от особенностей разных баз данных решение для генерирования подобных ключей,
// в этом прикладном интерфейсе предоставляется эффективный способ их извлечения.

// Если при вводе новой строки в таблицу автоматически генерируется ключ,
// его можно получить с помощью следующего кода:
    stmt.executeUpdate(insertStatement, Statement.RETURN_GENERATED_KEYS);
    ResultSet rs = stmt.getGeneratedKeys();
    if (rs.next()) {
        int key = rs.getInt(1);
        // ...
    }

// 5.6 Прокучиваемые и обновляемые результирующие наборы (Scrollable and Updatable Result Sets)

// Метод next() из интерфейса ResultSet позволяет
// последовательно перебирать строки в результирующем наборе,
// получаемом по запросу базы данных.

// Его очень удобно использовать для анализа полученных данных.
// Нередко пользователю требуется предоставить возможность
// для просмотра результатов выполнения запроса с переходом к предыдущей и следующей строке,
// как было показано на рис. 5.4 (resources/_293/pic_5_4.JPG).

// В Прокручиваемом (Scrollable) результирующем наборе
// можно перемещаться не только к предыдущим и последующим записям, но и на произвольную позицию.

// При просмотре результатов выполнения запроса у пользователей
// часто возникает потребность исправить, какие-нибудь данные.

// В обновляемом (Updatable) результирующем наборе можно видоизменять записи программно,
// чтобы автоматически обновить их в базе данных.


// page 325
// 5.6.1 Прокручиваемые результирующие наборы (Scrollable Result Sets)

// По умолчанию результирующие наборы не являются прокручиваемыми или обновляемыми.

// Для организации прокрутки результатов выполнения запроса
// необходимо получить объект типа Statement следующим образом:
    Statement stat = conn.createStatement(type, concurrecy);

// Для подготовленного оператора потребуется следующий вызов:
    PreparedStatement prepStat = conn.prepareStatement(command, type, concurrency);

// Допустимые значения параметров:
// - type
// - concurrency
// перечислены в табл. 5.6 и 5.7.

// Таблица 5.6. Значения параметра type, представленные константами из интерфейса ResultSet
// |-------------------------|----------------------------------------------------|
// |         Значение        |                     Описание                       |
// |-------------------------|----------------------------------------------------|
// | ТУРЕ_FORWARD_ONLY       | Без прокрутки (по умолчанию)                       |
// | ТУРЕ_SCROLL_INSENSITIVE | С прокруткой, но без учета изменений в базе данных |
// | ТУРЕ_SCROLL_SENSITIVE   | С прокруткой и с учетом изменений в базе данных    |
// |-------------------------|----------------------------------------------------|

// Таблица 5.7. Значения параметра concurrency, представленные константами из интерфейса ResultSet
// |-------------------|------------------------------------------------------------|
// |     Значение      |                     Описание                               |
// |-------------------|------------------------------------------------------------|
// | CONCUR_READ_ONLY  | Без редактирования и обновления базы данных (по умолчанию) |
// | CONCUR_UPDATAВLE  | С редактированием и обновлением базы данных                |
// |-------------------|------------------------------------------------------------|

// Выбирая эти значения, придется найти ответы на следующие вопросы:
// • Требуется ли сделать результирующий набор прокручиваемым?
//   Если этого не требуется, следует выбрать значение:
//   - ResultSet.TYPE_FORWARD_ONLY

// • Если требуется сделать результирующий набор прокручиваемым,
//   то должен ли он отражать те данные, которые были изменены в базе данных после выполнения запроса?
//   Здесь и далее предполагается, что установлен параметр:
//   - ResultSet.TYPE_SCROLL_INSENSITIVE
//   т.е. результирующий набор не реагирует на те изменения,
//   которые произошли в базе данных после выполнения запроса.

// • Требуется ли отредактировать результирующий набор и обновить базу данных?
//   Более подробно этот вопрос рассматривается в следующем разделе.

// Если требуется только прокрутка результирующего набора,
// но не редактирование его данных, это можно организовать следующим образом:
     Statement stat = conn.createStatement(ResultSet.TYPE_SCROLL_INSENSITIVE, ResultSet.CONCUR_READ_ONLY);

// Теперь можно прокручивать все результирующие наборы,
// возвращаемые при вызовах приведенного ниже метода.
// Получаемый в итоге результирующий набор содержит курсор, устанавливаемый на текущей позиции.
    ResultSet rs = stat.executeQuery(query)

// Не все драйверы баз данных поддерживают прокручиваемые или обновляемые результирующие наборы.
// Методы:
// - supportsResultSetТype()
// - supportsResultSetConcurrency()
// из интерфейса DatabaseМetaData сообщают о типах и режимах параллельной обработки,
// которые поддерживаются в конкретной базе данных с помощью определенного драйвера.

// Даже если в базе данных поддерживаются результирующие наборы во всех описанных режимах,
// то в некоторых запросах нельзя получить результирующий набор со всеми запрашиваемыми свойствами.

// Например, результат выполнения сложного запроса может оказаться необновляемым.

// -> метод executeQuery() возвращает результирующий набор типа ResultSet с меньшими возможностями,
// вводя предупреждение типа SQLWarning в объект соединения.

// Способ извлечения предупреждений представлен в разделе 5.4.З.

// С другой стороны, для выявления конкретного режима работы результирующего набора
// можно вызвать методы из интерфейса ResultSet:
// - getТype()
// - getConcurrency()

// Нужно выяснить конкретный режим работы результирующего набора.

// Если попытаться выполнить неподдерживаемую в нем операцию, например, вызвать метод previous()
// для непрокручиваемого результирующего набора, это неизбежно приведет к исключению типа SQLException.

// Прокрутка организуется очень просто.
// Для перехода к предыдущим записям в результирующем наборе
// служит приведенная ниже конструкция.
    if (rs.previous()) {
        // ...
    }

// Метод previous() - возвращает логическое значение true,
//                    если курсор находится на конкретной строке в результирующем наборе,
//                    или логическое значение false,
//                    если курсор находится перед первой строкой.

//  метод relative() - возвращает логическое значение true только в том случае,
//                     если курсор устанавливается на конкретной строке.

// Для перемещения курсора на "n" строк вперед или назад вызывается следующий метод:
    rs.relative(n);

// При:
// - положительных значениях параметра "n" - курсор перемещается вперед
// - отрицательных значениях параметра "n" - курсор перемещается назад
// - нулевом значении параметра "n" - никакого перемещения не происходит

// Если попытаться переместить курсор за пределы текущего ряда строк,
// он расположится:
// - за последней строкой
        или
// - перед первой строкой
// в зависимости от знака в значении параметра "n".
// После этого метод relative() возвращает логическое значение false,
// а перемещение курсора прекращается.

// Курсор можно установить на конкретной строке под номером "n",
// вызвав следующий метод:
    s.absolute(n);

// Получить текущий номер строки "n" можно следующим образом:
    int currentRow = rs.getRow();

// Первая строка в результирующем наборе имеет номер 1.
// Если возвращаемое значение равно нулю, то курсор находится не на конкретной строке,
// а за последней строкой
//      или
// перед первой строкой

// Для установки курсора:
// на первой или последней строке,
// перед первой или за последней строкой
// результирующего набора предусмотрены удобные методы:
// - first()
// - last()
// - beforeFirst()
// - afterLast()

// Для проверки расположения курсора на одной из этих позиций удобные методы:
// - isFirst()
// - isLast()
// - isBeforeFirst()
// - isAfterLast()

// Все рутинные операции, связанные с кешированием данных,
// получаемых по запросу, выполняются драйвером базы данных.


// Дополнительно:
// Прокручиваемые (Scrollable) и обновляемые (Updatable) результирующие наборы
// - это дополнительные функциональные возможности интерфейса ResultSet в JDBC,
// которые расширяют базовую функциональность по работе с результатами запросов.

// Прокручиваемые результирующие наборы (Scrollable Result Sets):

// 1. Прокручиваемый ResultSet
//    Обычный ResultSet позволяет двигаться только вперед с помощью метода next().
//    Прокручиваемый ResultSet дает возможность перемещаться по строкам результата не только вперед,
//    но и назад, а также переходить на произвольную позицию.

//    Для этого используются методы:
//    - previous()
//    - first()
//    - last()
//    - absolute(int row)
//    - relative(int rows)
//    и другие.

//    Это удобно когда нужно предоставить пользователю возможность просматривать результаты запроса
//    с навигацией вперед-назад и переходом на определенную строку.

//    Чтобы создать прокручиваемый ResultSet,
//    нужно передать параметр типа результата в метод
//    - createStatement()
//    - prepareStatement()
//    например:
       Statement stmt = connection.createStatement(ResultSet.TYPE_SCROLL_INSENSITIVE, ResultSet.CONCUR_READ_ONLY);
//    Обновляемые результирующие наборы (Updatable Result Sets):

// 2. Обновляемый ResultSet
//    Обычный ResultSet предоставляет доступ только для чтения данных, полученных из БД.

//    Обновляемый ResultSet позволяет:
//    - читать данные
//    - изменять строки
//    - добавлять строки
//    - удалять строки
//    прямо через объект ResultSet.

//    Для редактирования значений в текущей строке ResultSet
//    предоставляет методы:
//    updateXXX()
//    *где - XXX разные типы данных

//    Пример:
//    updateString(int columnIndex, String x).

//    После изменения значений, нужно вызвать метод updateRow() для сохранения изменений в БД,
//    Для добавления новой строки используется метод moveToInsertRow(),
//    затем заполняются значения через updateXXX(),
//    далее вызывается insertRow().

//    Для удаления текущей строки существует метод deleteRow().
//    Чтобы создать обновляемый ResultSet,
//    нужно указать соответствующие параметры типа результата и параллелизма:
       Statement stmt = connection.createStatement(ResultSet.TYPE_SCROLL_SENSITIVE, ResultSet.CONCUR_UPDATABLE);

// Прокручиваемые и обновляемые ResultSet предоставляют гибкие возможности
// для навигации по результатам запроса и их редактирования
// без необходимости писать отдельные запросы на изменение данных в БД.

// Это может быть полезно для реализации пользовательских интерфейсов
// с просмотром и редактированием данных, полученных из БД.

// Следует учитывать, что не все СУБД и драйверы JDBC
// полностью поддерживают все возможности прокручиваемых и обновляемых ResultSet.

// Перед их использованием, нужно проверить документацию и провести тестирование в конкретной среде.


// Прокручиваемые и обновляемые результирующие наборы могут быть полезны в различных сценариях,
// где требуется удобная навигация по данным и их редактирование.

// 1. Просмотр и редактирование данных в пользовательском интерфейсе:
//    Пример:
//    Разрабатка приложение для управления списком студентов.
//    Можно использовать прокручиваемый ResultSet
//    для отображения списка студентов в пользовательском интерфейсе,
//    позволяя пользователю перемещаться вперед и назад по списку,
//    а также переходить на определенного студента.

//    Если пользователю нужно отредактировать данные студента, можно использовать обновляемый ResultSet,
//    чтобы изменить значения прямо в ResultSet и сохранить изменения в базе данных.

//    Пример:
        // Получение прокручиваемого и обновляемого ResultSet
        Statement stmt = connection.createStatement(ResultSet.TYPE_SCROLL_SENSITIVE, ResultSet.CONCUR_UPDATABLE);
        String query = "SELECT * FROM students";
        ResultSet resultSet = stmt.executeQuery(query);

        // Переход к определенной строке (например, третьей)
        resultSet.absolute(3);

        // Изменение значения в текущей строке
        String newEmail = "newstudent@example.com";
        resultSet.updateString("email", newEmail);
        resultSet.updateRow();

        // Переход к следующей строке
        resultSet.next();

        // Изменение значения в текущей строке
        String newPhone = "1234567890";
        resultSet.updateString("phone", newPhone);
        resultSet.updateRow();
        // ...

        resultSet.close();
        stmt.close();

//    В этом примере:
//    а. получаем прокручиваемый и обновляемый ResultSet из запроса SELECT
//    б. перемещаемся к определенной строке с помощью метода absolute()
//    в. изменяем значение email с помощью updateString()
//    г. вызываем updateRow() для сохранения изменений
//    д. переходим к следующей строке
//    е. обновляем значение phone аналогичным образом

// 2. Пакетное обновление данных:
//    Пример:
//    Нужно обновить определенное поле для множества записей в таблице на основе некоторого условия.
//    Вместо выполнения отдельного запроса UPDATE для каждой записи,
//    можно использовать обновляемый ResultSet.

//    Можете выполнить запрос SELECT с условием, получить обновляемый ResultSet,
//    пройтись по строкам, изменить необходимые значения с помощью методов updateXXX()
//    и вызвать updateRow() для каждой измененной строки.

//    Это может быть более эффективно, чем выполнение множества отдельных запросов UPDATE.

// 3. Обработка больших объемов данных:
//    Если вам нужно обработать большой объем данных из базы данных,
//    использование прокручиваемого ResultSet может быть более эффективным,
//    чем загрузка всех данных в память сразу.

//    Можно получить прокручиваемый ResultSet и обрабатывать данные порциями,
//    перемещаясь вперед и назад по результатам.

//    Это позволяет обрабатывать данные в потоковом режиме, не потребляя большой объем памяти.

// 4. Реализация пользовательских отчетов:
//    При создании пользовательских отчетов,
//    где пользователь может просматривать и фильтровать данные,
//    прокручиваемый ResultSet может быть полезен для навигации по результатам отчета.

//    Пользователь может перемещаться вперед и назад по страницам отчета,
//    переходить на определенную страницу или строку.
//    Это улучшает удобство использования и интерактивность отчетов.

// Применение прокручиваемыех (scrollable) и обновляемыех (updatable) ResultSet
// при работе с картографией и позиционированием.

// Примеры:
// 1. Просмотр больших наборов геоданных:
//    Если есть большой объем геопространственных данных (координаты, описания местоположений и т.д.),
//    хранящийся в базе данных, то использование прокручиваемого ResultSet
//    позволяет эффективно просматривать данные порциями,
//    без необходимости загружать весь набор целиком в память.
//    Это экономит ресурсы при работе с большими картами.

// 2. Обновление координат в реальном времени:
//    С помощью обновляемого ResultSet можно
//    динамически изменять координаты объектов прямо в наборе данных,
//    не перезагружая его целиком.
//    Это удобно, если позиции отслеживаемых объектов меняются
//    в реальном времени
//    Пример:
//    - движущийся транспорт

// 3. Пакетная обработка геоданных:
//    Scrollable ResultSet дает возможность последовательно обрабатывать данные
//    о множестве географических объектов, производя необходимые вычисления и анализ.
//    Это эффективнее, чем загружать данные целиком.

// 4. Интеграция с ГИС и картографическими сервисами:
//    Данные из ResultSet можно передавать в различные ГИС
//    и сервисы картографии для отображения на картах.
//    Прокручиваемость и обновляемость ResultSet облегчают синхронизацию
//    между базой данных и картографическим ПО.

// 5. Выборка данных по регионам:
//    Scrollable ResultSet позволяет выбирать геоданные,
//    соответствующие определенной пространственной области,
//    без лишней загрузки данных по остальной карте.

// -> Прокручиваемые и обновляемые ResultSet предоставляют гибкие возможности для работы с геоданными:
// - анализ
// - изменение
// - отображение
// на картах.

// Они повышают производительность и эффективность в картографических и ГИС приложениях,
// особенно при работе с большими и динамическими наборами геопространственной информации.


// page 327
// 5.6.2 Обновляемые результирующие наборы (Updatable Result Sets)

// Если требуется отредактировать данные, полученные по запросу в результирующем наборе,
// а также автоматически обновить базу данных, такой набор следует сделать обновляемым.

// Обновляемые результирующие наборы совсем не обязательно должны быть прокручиваемыми.

// Если требуется предоставить пользователю возможность редактировать данные,
// то они должны допускать и прокрутку.

// Пример кода для получения обновляемого результирующего набора:
    Statement stat = conn.createStatement(
                            ResultSet.TYPE_SCROLL_INSENSITIVE,
                            ResultSet.CONCUR_UPDATABLE);

// для этой цели указывается - "ResultSet.CONCUR_UPDATABLE".

// Результирующие наборы, возвращаемые методом executeQuery(),
// становятся в итоге обновляемыми.

// Обновляемый результирующий набор возвращается не по всем запросам.

// Если в запросе предполагается соединение нескольких таблиц,
// его результат не всегда может быть обновляемым.

// Если в запросе предполагается обращение к одной таблице
// или соединение нескольких таблиц по их первичным ключам,
// то следует ожидать, что получаемый в итоге результирующий набор окажется обновляемым.

// Чтобы выяснить, является ли результирующий набор обновляемым,
// следует вызвать метод getConcurrency() из интерфейса ResultSet.

// Задача:
// Требуется повысить цену на некоторые книги, но отсутствует единый критерий,
// который можно было бы использовать для этого в команде UPDATE.

// Решение:
// Придется перебрать в цикле все книги и изменить цены по произвольным условиям:
    String query = "SELECT * FROM Books";
    ResultSet rs = stat.executeQuery(query);
    while (rs.next()) {
        if (...) {
            double increase = ...
            double price = rs.getDouble("Price");
            rs.updateDouble("Price", price + increase);
            rs.updateRow(); // непременно вызвать метод updateRow() после обновления полей в таблице
        }
    }

// Для всех типов данных SQL предусмотрены соответствующие формы метода updateXxx(),
// Пример:
// - updateDouble()
// - updateString()
// и т.д.

// Как и при вызове различных форм метода getXxx(),
// в качестве параметров данного метода могут быть указаны:
// - номер
// - имя столбца,
// а затем новое значение поля.

// Первый параметр метода updateXxx()
// обозначает номер столбца в результирующем наборе,
// где он может отличаться от номера столбца в базе данных.

// В методе updateXxx() изменяется только значения полей в текущей строке результирующего набора,
// а не в самой базе данных.

// Для обновления всех полей отредактированной строки в базе данных
// следует вызвать метод updateRow().

// Если же переместить курсор к следующей строке, не вызывая метод updateRow(),
// все обновления предыдущей строки в результирующем наборе будут отменены,
// поскольку они не были переданы базе данных.

// Для отмены обновлений текущей строки в базе данных следует вызвать метод cancelRowUpdates().

// Демонстрация порядока внесения изменений в существующей строке:

// 1. Для создания новой строки в базе данных нужно сначала вызвать метод moveToInsertRow(),
//    переместив тем самым курсор на специальную позицию, называемую строкой вставки.

// 2. Затем новая строка создается на данной позиции с помощью метода updateXxx().

// 3. Для передачи новой вставляемой строки в базу данных вызывается метод insertRow().

// 4. По окончании вставки вызывается метод moveToCurrentRow(),
//    чтобы переместить курсор назад на ту позицию,
//    которую он занимал до вызова метода moveToInsertRow().

// Каким образом весь этот процесс реализуется непосредственно в коде:
    rs.moveToInsertRow();
    rs.updateString("Title", title);
    rs.updateString("ISBN", isbn);
    rs.updateString("Publisher_Id", pubid);
    rs.updateDouble("Price", price);
    rs.insertRow();
    rs.moveToCurrentRow();

// Однако конкретное расположение новых данных в результирующем наборе
// или базе данных не поддается непосредственному управлению из прикладного кода.
// *Нельзя повлиять на то, где будут добавлены данные, в результирующем наборе или БД.

// Если не указать конкретное значение для столбца в строке вставки,
// на этом месте окажется пустое значение NULL.

// Если на столбец наложено ограничение NOT NULL,
// то сгенерируется исключение и строка не будет вставлена.

// Можно удалить ту строку, на которой установлен курсор:
    rs.deleteRow();
// Данный метод, немедленно удаляет строку как из результирующего набора, так из базы данных:

// Методы из интерфейса ResultSet:
// - updateRow()
// - insertRow()
// - deleteRow()
// предоставляют те же возможности, что и операторы языка SQL:
// - UPDATE
// - INSERT
// - DELETE

// На Java вызов методов довольно привычен,
// поэтому можно использовать данный подход
// составлению запросов из операторов SQL.

// При неаккуратной обработке обновляемых результирующих наборов
// можно получить совершенно неэффективный код.

// Часто выполнение оператора UPDATE оказывается намного более эффективным,
// чем составление запроса и просмотр результирующего набора.

// Обработку обновляемых результирующих наборов имеет смысл
// организовывать в диалоговых прикладных программах,
// где пользователь может вносить произвольные изменения.

// Для внесения заранее программируемых изменений более подходящим оказывается оператор UPDAТE.

// В версии JDBC 2 были внедрены дополнительные усовершенствования в результирующие наборы,
// в том числе возможность обновлять результирующие наборы самыми последними данными,
// если они были изменены при другом, параллельном соединении с базой данных.

// В версии JDBC 3 было внедрено еще одно усовершенствование,
// определяющее режим работы результирующих наборов при фиксации транзакции.

// Но эти дополнительные возможности здесь не рассматриваются,
// поскольку они выходят за рамки введения в базы данных.

// Дополнительными сведениями о JDBC 2-3 книге:
// JОВС АРI Tutorial and Reference, Third Edition
// Мэйдена Фишера, Джона Эллиса и Джонатана Брюса Мауdепе Fisheг, Jоп Ellis, Joпathaп Вгuсе;
// издательство Addisoп-Wesley, 2003 г.,
// а также к документации, описывающей спецификацию прикладного интерфейса JDBC
// и доступной для загрузки по адресу:
     https://download.oracle.com/otndocs/jcp/jdbc-4_2-mrel2-spec/


// page 331
// 5.7 Нвбор строк (Row Sets)

// Прокручиваемые результирующие наборы предлагают большие возможности,
// но у них есть недостатки.

// В течение всего периода взаимодействия с пользователем
// должно быть установлено соединение с базой данных.
// Но ведь пользователь может отлучиться на длительное время,
// а между тем установленное соединение будет напрасно потреблять сетевые ресурсы.

// В подобной ситуации целесообразно использовать набор строк.
// Интерфейс RowSet расширяет интерфейс ResultSet,
// но набор строк не должен быть привязан к соединению с базой данных.

// Наборы строк применяются и в том случае,
// если требуется перенести результаты выполнения запроса на другой уровень
// сложного приложения или на другое устройство, например на мобильный телефон.

// Перенести результирующий набор нельзя, поскольку он связан с подключением к базе данных,
// кроме того, структура данных может иметь довольно крупные размеры.


// page 331
// 5.7.1 Построение наборов строк (Constructing Row Sets)

// Интерфейсы, входящие в пакет javax.sql.rowset и расширяющие интерфейс RowSet:

// 1. Интерфейс CachedRowSet - позволяет выполнять некоторые операции при отсутствии соединения.
//    Кешируемые наборы строк рассматриваются в следующем разделе.

// 2. Интерфейс WebRowSet - представляет кешируемый набор строк,
//    который может быть сохранен в ХМL-файле.
//    Сам ХМL-файл может быть передан другому компоненту приложения
//    и открыт с помощью другого объекта типа WebRowSet.

// 3. Интерфейсы FilteredRowSet и JoinRowSet поддерживают легковесные операции с наборами строк,
//    равнозначные таким командам SQL, как SELECT и JOIN.
//    Эти операции выполняются только над данными, содержащимися в наборе строк,
//    для чего не требуется устанавливать соединение с базой данных.

// 4. Интерфейс JdbcRowSet является тонкой оболочкой для интерфейса ResultSet,
//    вводя удобные методы из интерфейса RowSet.

// В версии java 7 появился стандартный способ получения набора строк
// с помощью приведенных ниже методов.

// Аналогичные методы имеются для получения наборов строк других типов.
    RowSetFactory factory = RowSetProvider.newFactory();
    CachedRowSet crs = factory.createCachedRowSet();


// page 332
// 5.7.2 Кешируемые наборы строк (Cached Row Sets)

// Кешируемый набор строк содержит данные из результирующего набора.
// Интерфейс CachedRowSet расширяет интерфейс ResultSet,
// -> кешируемым набором строк можно пользоваться точно так же, как и результирующим.

// Наборы строк имеют существенное преимущество,
// позволяющее отключиться от базы данных и продолжать работать с набором строк.

// Исходный код программы представлен в листинге 5.4 (ru/sectorsj/_337_view/ViewDB.java)
// упрощающий создание диалоговых приложений.

// - при получении команды от пользователя
// - осуществляется подключение к базе данных
// - выполняется запрос
// - результаты размещаются в наборе строк
// - после чего производится отключение от базы данных

// Кешируемый набор строк позволяет видоизменить содержащиеся в нем данные.
// Результаты подобных изменений не отражаются в базе данных немедленно.
// Чтобы принять накопленные изменения, необходимо выполнить явный запрос.

// Кешируемый набор строк типа CachedRowSet повторно подключается к базе данных
// и выполняет операторы SQL для записи изменений в базу данных.

// Кешируемый набор строк типа CachedRowSet заполняется данными
// из результирующего набора следующим образом:
    ResultSet result = ...;
    RowSetFactory factory = RowSetProvider.newFactory();
    CachedRowSet crs = factory.createCachedRowSet();
    crs.populate(result);
    conn.close();   // теперь можно отключиться от базы данных

// Кешируемому набору строк типа CachedRowSet
// можно предоставить возможность автоматически подключиться к базе данных.

// Для этого сначала задаются следующие параметры базы данных:
    crs.setURL("jdbc:derby://localhost:1527/COREJAVA");
    crs.setUsername("dbuser");
    crs.setPassword("secret");

// Составляется оператор запроса с любыми параметрами:
    crs.setCommand("SELECT * FROM Books WHERE Publisher = ?");
    crs.setString(1, publisherName);

// - набор строк заполняется результатами запроса
// - осуществляется подключение к базе данных
// - выполняется запрос
// - заполняется набор строк
// - производится отключение от базы данных
    crs.execute();

// Если полученный результат запроса имеет слишком большой объем,
// его можно и не полностью вводить в набор строк.
// Пользователи прикладной программы, веротяно просмотрят лишь некоторые строки.

// Определение размеров выводимой страницы:
    CachedRowSet crs = ...;
    crs.setCommand(command);
    crs.setPageSize(20);
    // ...
    crs.execute();

// В итоге будет доступно только 20 строк.
// Для получения следующей порции строк, достаточно вызвать метод:
    crs.nextPage();

// Для просмотра и видоизменения набора строк служат те же методы,
// что и для обращения с результирующим набором.

// Если изменить содержимое набора строк, для записи изменений в базу данных
// необходимо сделать один из следующих вызовов:
    crs.acceptChanges(conn);
//           или
    crs.acceptChanges();

// Второй вариант вызова метода acceptChanges() действует только в том случае,
// если предоставить для набора строк все сведения,
// необходимые для подключения к базе данных:
//  - URL
//  - имя пользователя
//  - пароль

// Не все результирующие наборы являются обновляемыми (раздел 5.6.2).

// Наборы строк, содержащие результаты сложных запросов,
// не позволяют записывать изменения в базу данных.

// Если же набор строк содержит данные только из одной таблицы,
// то никаких затруднений при их записи в базу данных не возникает.

// Если заполнить набор строк данными из результирующего набора,
// то набору строк не будет известно имя обновляемой таблицы.

// Придется специально указать имя таблицы, вызвав метод:
    setTableName()

// Если данные в базе изменились с того момента, как набор строк был заполнен ими,
// то возникают дополнительные затруднения, связанные с несоответствием данных.

// В базовой реализации проверяется:
//  - совпадают ли исходные значения из набора строк

// page 345
// 5.9 Транзакции (Transactions)

// Группа команд может быть оформлена в виде транзакции,
// которая может быть зафиксирована после успешного выполнения всех операторов SQL или откачена,
// если при выполнении хотя бы одного из операторов произойдет какая-нибудь ошибка.

// Основной причиной для группирования операторов в транзакции
// служит сохранение целостности базы данных.

// Задание:
// Требуется перевести денежные средства с одного банковского счета на другой.

// Решение:
// Нужно одновременно снять нужную сумму денег с одного счета и пополнить ею другой счет.

// Если после снятия суммы денег с одного счета,
// но перед пополнением другого произойдет системная ошибка,
// то операция с первым счетом должна быть отменена.

// Операторы обновления базы данных могут быть сгруппированы в одну транзакцию.

// Если транзакция завершается полностью, то возможна ее фиксация.

// Если транзакция не завершается полностью из-за каких-нибудь ошибок,
// то производится ее откат, т.е. отменяются все изменения в базе данных,
// которые выполнялись после предыдущей зафиксированной транзакции.


// page 345
// 5.9.1 Программирование транзакций средствами JDBC (Programming Transactions with JDBC)

// По умолчанию соединение с базой данных находится в режиме автоматической фиксации,
// т.е. результат выполнения каждого оператора SQL фиксируется в базе данных
// после успешного завершения этой команды.

// Как только оператор будет зафиксирован, откатить его уже нельзя.

// Для отключения режима автоматической фиксации можно вызвать следующий метод:
    conn.setAutoCommit(false);

// Отключив автоматическую фиксацию, можно приступать к созданию объекта типа Statement обычным образом:
    Statement stat = conn.createStatement();

// Метод executeUpdate() вызывается нужное количество раз:
    stat.executeUpdate(command1);
    stat.executeUpdate(command2);
    stat.executeUpdate(command3);
    // ...

// Если все эти операторы завершатся успешно, то результаты их выполнения фиксируются.
// Для этого вызывается метод commit() следующим образом:
    conn.commit();

// Если при выполнении любого из этих операторов произойдет ошибка,
// то производится откат всей транзакции.
// Для этого вызывается метод rollback(), следующим образом:
    conn.rollback();

// При этом автоматически отменяются все операторы, выполнявшиеся после фиксации последней транзакции.

// Откат производится, если при выполнении транзакции генерируется исключение типа SQLException.
